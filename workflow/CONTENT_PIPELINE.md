# 内容分流流水线

发布链路固定，写作结构不固定。

固定发布链路：

```text
素材 -> 草稿 -> 核验 -> 封面和配图 -> WeMD -> HTML -> 公众号草稿箱
```

写作前先判断内容类型，再选择结构、素材和语气。

## 总入口

每次开始写公众号文章，先确认：

1. 当前任务是写稿、改稿、做封面、WeMD 审稿，还是推草稿箱。
2. 文章类型是什么：教程、工具介绍、产品介绍、案例复盘、观点经验、活动通知、资料整理或其它明确类型。
3. 素材来自哪里：本地文档、截图、网页链接、代码、表格、访谈记录、用户口述或已有草稿。
4. 哪些事实必须核验，哪些位置需要保留截图占位。

信息足够时直接写完整稿，不停在大纲。

## 内容类型

### A. 教程实操

适用：工具使用、开发步骤、部署流程、排版流程、自动化流程。

重点：适用场景、准备条件、操作步骤、每一步结果、常见卡点。

### B. 工具或产品介绍

适用：软件、插件、平台、AI 工具、效率工具。

重点：解决什么具体问题、如何上手、关键功能截图、真实体验、局限和注意点。

### C. 案例复盘

适用：一次项目过程、一次发布实践、一次自动化流程、一次运营动作。

重点：背景、决策、过程、遇到的问题、怎么解决、哪些经验可以复用。

### D. 观点或经验文

适用：行业看法、个人经验、方法论、选型判断。

重点：一个明确判断、两到四个支撑理由、真实例子、反面边界、可执行建议。

### E. 活动或说明文

适用：活动说明、版本发布、资料整理、社群通知、开放样例。

重点：开放什么、为什么做、能看到哪些真实内容、如何使用、后续注意什么。

### F. 项目或系统拆解

适用：网站、小程序、管理后台、数据系统、自动化脚本或其它可展示实现过程的项目。

重点：需求背景、模块结构、关键流程、页面或运行截图、实现细节、可复用部分。

这类文章也要从素材出发。没有真实截图、代码路径或运行结果时，不要写成已经完成。

## 质量门

成稿后检查：

1. 事实门：结论能回到素材、截图、代码、数据、链接或用户口述。
2. 类型门：结构符合内容类型，没有套错模板。
3. 口吻门：已按 `humanizer-zh` 过一遍，没有空泛赞美、硬转折、夸大承诺、机械总结和无证据判断。
4. 图片门：封面和正文图都有来源，缺图的位置用清晰占位。
5. 微信门：标题、摘要、封面、图片、HTML 和底部设置都能进入预览检查。
6. 确认门：用户确认后，才推公众号草稿箱。

## 去 AI 化

每次生成或大改正文后，都按 `humanizer-zh` 做一遍人工化编辑：

- 删除“赋能、打造、助力、深度解析、全方位、显著提升”等空词。
- 删掉“首先、其次、最后”撑出来的假结构。
- 删除“不是……而是……”“不仅……还……”这类模板化转折。
- 删掉没有素材、截图、数据、链接或用户口述支撑的判断。
- 少用夸张总结，多写真实动作和具体细节。
- 句子短一点，段落轻一点，移动端读起来不要喘不过气。
